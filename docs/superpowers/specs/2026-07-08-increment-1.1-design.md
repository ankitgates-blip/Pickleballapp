# Increment 1.1 Design — Bare Tournament Core

Status: Draft, pending user review.
Parent doc: [PLAN.md](../../../PLAN.md) (Phase 1, Increment 1.1).

## Goal

Smallest usable slice that replaces a spreadsheet for running one round-robin doubles
tournament at Pickle Turf: create a tournament, paste in a roster, manually pair teams,
auto-generate a round-robin schedule, enter scores, see live standings — and share a
read-only link so players can check the bracket on their phones without logging in.

## Scope decisions (resolved via brainstorming, 2026-07-08)

- **Surface**: organizer web dashboard only (Next.js). No player mobile app yet — players
  view a public, unauthenticated, read-only link instead. Player app arrives in Phase 1.5.
- **Bracket format**: round robin only (single elimination deferred — no seeding/bye logic
  to build for 1.1 beyond round robin's own bye handling).
- **Team formation**: doubles, manually paired by the organizer. Auto team shuffle
  (Increment 1.2) is a distinct, later upgrade to this same team-pairing step.
- **Scoring**: one score per match (e.g. 11–7), not best-of-N games. Target score and
  win-by margin are configurable per tournament but don't add data-model complexity.
- **Auth**: Supabase auth (email/password or phone OTP) for the organizer from day one,
  even though there's only one organizer today — avoids retrofitting auth once co-directors
  (Phase 2.5) need their own logins, and keeps tournament data private by default.
- **Data access pattern**: dashboard calls Supabase directly via its JS SDK; Postgres
  row-level security (RLS) enforces access control. No custom Next.js API route layer.
  Round-robin generation and standings computation run as Postgres functions (RPC), not
  a bespoke backend service — keeps this a "modular monolith on Supabase," per PLAN.md §4.

## Screens

| Screen | Purpose |
|---|---|
| Login | Organizer signs in (Supabase Auth) |
| Tournament list | Organizer's tournaments; "+ New Tournament" |
| Create tournament | Name, date, venue (fixed to Pickle Turf for now), target score, win-by |
| Roster entry | Paste/type player names, one per line → creates `players` rows scoped to this tournament |
| Team pairing | Manually pair two roster entries into a `teams` row |
| Generate bracket | Button triggers the round-robin RPC, populating `matches` |
| Score entry | List of matches; tap one, enter score, mark complete; standings recompute |
| Standings / bracket view | Live standings + match list; "Copy public link" button |
| Public read-only view | Same standings/schedule, no login, scoped by tournament ID in the URL |

## Data model (Postgres via Supabase, additive-only)

```
venues       (id, name)                                        -- seeded: "Pickle Turf"
organizers   (id, auth_user_id, name)
tournaments  (id, venue_id, organizer_id, name, date, target_score, win_by)
players      (id, tournament_id, name, claimed_by_user_id nullable)
teams        (id, tournament_id, player_1_id, player_2_id)
matches      (id, tournament_id, team_a_id, team_b_id, court, score_a, score_b,
              status: pending|complete)
```

Notes:
- `players.claimed_by_user_id` is nullable from day one so the roster-claiming behavior
  (PLAN.md §6) needs no later migration.
- `matches.score_a`/`score_b` are plain integers (single game per match). A later increment
  can add a child `games` table for best-of-N without altering this table — additive, per
  PLAN.md §4's migration rule.
- RLS: an organizer can only read/write tournaments where `organizer_id` matches their
  `auth_user_id`-linked row, and everything nested under it (players/teams/matches via
  join). The public view is a separate unauthenticated RPC/view scoped strictly by
  tournament ID — returns only names/scores/standings, no PII beyond player names.

## Core logic (Postgres RPC functions)

- `generate_round_robin(tournament_id)`: given all `teams` for a tournament, creates every
  unique pairing as a `matches` row. Handles odd team counts by giving one team a bye per
  round (tracked, not silently dropped — the organizer should see who has a bye).
- `recompute_standings(tournament_id)`: derives wins/losses/points-for/points-against per
  team from completed `matches`; used to render the standings screen. Recomputed on every
  score entry rather than incrementally, since match counts per tournament are small.

## Edge cases

- Odd number of teams → bye handling in the round-robin generator (see above).
- Duplicate player names (two "Mike"s) → allowed as distinct roster rows; UI shows a
  warning so the organizer doesn't mis-pair teams.
- Editing a score after a match is marked complete → allowed; standings recompute.
- Fewer than 2 teams when "Generate bracket" is clicked → blocked with a clear message.

## Testing

- Unit: `generate_round_robin` (correct match count for N teams, bye handling for odd N),
  `recompute_standings` (wins/losses/points from a fixed set of match scores).
- Integration: create tournament → roster → pair teams → generate bracket → enter scores →
  verify standings end to end.
- Manual acceptance: a real round-robin tournament run live at Pickle Turf, per the
  incremental-delivery rule — 1.1 isn't "done" until that happens and the organizer
  confirms it worked.

## Out of scope for 1.1 (explicitly deferred)

- Auto team shuffle (1.2), self-registration (1.3), payments/waivers (1.4), player app
  and push notifications (1.5), venue theming (1.6), organizer dashboard analytics beyond
  basic standings (1.7). See PLAN.md §3 for the full sequence.
