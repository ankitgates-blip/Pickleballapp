# Increment 1.2 Design — Auto Team Shuffle

Status: Approved, pending spec review before implementation plan.
Parent doc: [PLAN.md](../../../PLAN.md) (Phase 1, Increment 1.2).

## Goal

Replace the all-manual team pairing step from Increment 1.1 with a one-click option:
randomly shuffle all currently-unpaired roster players into balanced-count doubles teams,
while still allowing manual pairing for anyone who wants to hand-pick specific pairs. Also
add the ability to undo a pairing (auto-shuffled or manual) so mistakes and unwanted
shuffles can be corrected without deleting the whole roster.

## Scope decisions (resolved via brainstorming, 2026-07-09)

- **Scoped to shuffle only** — the "smart format recommender" that PLAN.md bundles with
  auto team shuffle in Increment 1.2 is deferred. With only round-robin built so far,
  there is nothing yet for a format recommender to choose between; it stays deferred until
  single-elimination or pool play exist (see PLAN.md §3, Phase 2+).
- **"Balanced" means count-balanced, not skill-balanced** — players currently have no
  skill/rating field captured anywhere in the data model. True skill-balanced shuffling
  needs that data to exist first (see PLAN.md §3.2/Phase 3), which is out of scope here.
  For 1.2, "shuffle" means: randomly pair up all unpaired players into teams of 2, with
  one player left over if the unpaired count is odd (matching how manual pairing already
  behaves with an odd roster).
- **Manual pairing is not removed** — shuffle and manual pairing coexist on the same Teams
  page. An organizer can shuffle everyone at once, manually pair a few and shuffle the
  rest, or manually pair everyone as before.
- **Undo via "Remove" on a team, not a preview/confirm step** — rather than showing a
  shuffle preview before committing, one click immediately creates the teams. To undo,
  each team row gets a "Remove" button that deletes that team and returns both players to
  the unpaired pool. "Re-shuffling" is: remove the teams you don't like, click shuffle
  again. This also fixes a standing gap from 1.1 — there was previously no way to undo a
  manual pairing mistake at all.

## Logic (`lib/tournament/shuffle.ts`, plain TypeScript — same pattern as `roundRobin.ts`)

- `shuffleIntoTeams(playerIds: string[], rng: () => number = Math.random): Array<{ player1Id: string; player2Id: string }>`:
  Fisher-Yates shuffle of the input array using the injectable `rng`, then pairs up
  consecutive elements. If `playerIds.length` is odd, the last player is left unpaired
  (not included in any returned team) — the calling UI already shows unpaired players
  separately, so this needs no special handling beyond "don't include them."
  The `rng` parameter defaults to `Math.random` for real use but lets unit tests inject a
  fixed sequence to assert deterministic pairing behavior instead of asserting on
  randomness itself (e.g. "every input ID appears in exactly one output pair or is left
  over," "no ID is paired with itself," "all pairs use valid input IDs").

## Data model

No schema change. `teams` (from Increment 1.1) already supports the operations needed:
- Insert: existing `teams_insert_own_tournament` RLS policy already covers bulk-inserting
  shuffle-generated teams the same way manual pairing does.
- Delete: existing `teams_delete_own_tournament` RLS policy (added in Increment 1.1's
  migration but not yet used by any UI) covers the new "Remove" button.

## UI changes (`app/tournaments/[id]/teams/`)

- New server action `shuffleRemaining(tournamentId)`: reads all unpaired players (same
  query the page already does to compute `unpairedPlayers`), calls `shuffleIntoTeams` on
  their IDs, bulk-inserts the resulting pairs as `teams` rows.
- New server action `removeTeam(tournamentId, teamId)`: deletes the `teams` row by id.
  Both players automatically reappear in "Unpaired players" on next render, since that
  list is derived from "players not referenced by any team row" — no extra bookkeeping.
- Page changes: a "Shuffle Remaining Players" button next to the existing manual-pair
  form (disabled/hidden if fewer than 2 unpaired players); each row in the "Teams" list
  gets a small "Remove" button.

## Edge cases

- Shuffling with 0 or 1 unpaired players: button does nothing / is disabled — no team can
  be formed from fewer than 2 players.
- Shuffling with an odd number of unpaired players: pairs as many as possible, one player
  remains in the unpaired pool (visible, not silently dropped).
- Removing a team whose bracket/matches already exist (i.e., shuffle/pairing changed after
  "Generate Bracket" was already run on the old team set): out of scope for this increment
  — 1.1's bracket generation is a one-time action gated on team count, and this increment
  doesn't add any interaction between "Remove team" and already-generated matches. If an
  organizer removes a team after generating a bracket, the existing matches referencing
  that team are unaffected by this increment (a future increment could add bracket
  regeneration, but that's not requested here).

## Testing

- Unit (Vitest): `shuffleIntoTeams` — even count produces `n/2` pairs covering every input
  ID exactly once; odd count produces `(n-1)/2` pairs plus exactly one leftover ID; no pair
  contains the same ID twice; using a fixed injected `rng` produces a specific, assertable
  pairing (regression-safe without asserting on true randomness).
- Manual/browser verification: shuffle a roster of 6 into 3 teams, confirm all 6 appear
  paired and 0 remain unpaired; remove one shuffled team, confirm its 2 players return to
  the unpaired list; re-shuffle remaining unpaired players; confirm manual pairing still
  works alongside shuffle for any players left unpaired.

## Out of scope for 1.2 (explicitly deferred)

- Smart format recommender (needs multiple formats to exist first).
- Skill-level-balanced shuffling (needs a skill/rating field, not yet captured — see
  PLAN.md Phase 3.2).
- Bracket regeneration after teams change post-bracket-generation.
